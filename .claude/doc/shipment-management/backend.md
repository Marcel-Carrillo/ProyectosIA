```typescript
export type ShipmentStatus =
  | 'Pending'
  | 'Shipped'
  | 'InTransit'
  | 'Delivered'
  | 'Failed'
  | 'Returned';

export const SHIPMENT_TRANSITIONS: Record<ShipmentStatus, ShipmentStatus[]> = {
  Pending:   ['Shipped', 'Failed', 'Returned'],
  Shipped:   ['InTransit', 'Delivered', 'Failed', 'Returned'],
  InTransit: ['Delivered', 'Failed', 'Returned'],
  Delivered: [],
  Failed:    [],
  Returned:  [],
};

export function isValidShipmentTransition(
  from: ShipmentStatus,
  to: ShipmentStatus
): boolean {
  return SHIPMENT_TRANSITIONS[from]?.includes(to) ?? false;
}

export class Shipment {
  id?: number;
  customerOrderId: number;
  supplierOrderId: number | null;
  carrier: string | null;
  trackingNumber: string | null;
  trackingUrl: string | null;
  status: ShipmentStatus;
  shippedAt: Date | null;
  deliveredAt: Date | null;
  createdAt?: Date;
  updatedAt?: Date;
  // Optional relation summaries — populated only by findById
  customerOrder?: { id: number; orderNumber: string; status: string };
  supplierOrder?: { id: number; status: string } | null;

  constructor(data: {
    id?: number;
    customerOrderId: number;
    supplierOrderId?: number | null;
    carrier?: string | null;
    trackingNumber?: string | null;
    trackingUrl?: string | null;
    status?: string;
    shippedAt?: Date | null;
    deliveredAt?: Date | null;
    createdAt?: Date;
    updatedAt?: Date;
    customerOrder?: { id: number; orderNumber: string; status: string };
    supplierOrder?: { id: number; status: string } | null;
  }) {
    this.id = data.id;
    this.customerOrderId = data.customerOrderId;
    this.supplierOrderId = data.supplierOrderId ?? null;
    this.carrier = data.carrier ?? null;
    this.trackingNumber = data.trackingNumber ?? null;
    this.trackingUrl = data.trackingUrl ?? null;
    this.status = (data.status as ShipmentStatus) ?? 'Pending';
    this.shippedAt = data.shippedAt ?? null;
    this.deliveredAt = data.deliveredAt ?? null;
    this.createdAt = data.createdAt;
    this.updatedAt = data.updatedAt;
    this.customerOrder = data.customerOrder;
    this.supplierOrder = data.supplierOrder;
  }
}
```

### Imports needed
None (pure domain type — no framework or Prisma imports).

### Key implementation notes

- Mirrors `refund.ts` structure exactly.
- Optional `customerOrder` and `supplierOrder` fields are populated only when the infra `findById` performs an `include`. List results leave them `undefined`.
- Terminal states have empty arrays in `SHIPMENT_TRANSITIONS` — calling `isValidShipmentTransition` from a terminal state always returns `false`.

**`ShipmentRepository` class implements `IShipmentRepository`:**

`findAll(filters: ShipmentFilters): Promise<ShipmentListResult>`
- Default `page = 1`, default `pageSize = 20`, cap `pageSize` at 100.
- Build `Prisma.ShipmentWhereInput` from filters: `customerOrderId`, `supplierOrderId`, `status`.
- Use `prisma.$transaction([findMany, count])`.
- `orderBy: { createdAt: 'desc' }`.
- Use `shipmentListSelect` (no includes — performance).
- Return `{ items: rows.map(mapShipment), total, page, pageSize }`.

`findById(id: number): Promise<Shipment | null>`
- `prisma.shipment.findUnique({ where: { id }, select: shipmentListSelect, include: shipmentDetailInclude })`.
- Note: do NOT mix `select` and `include` — use `include` alone for detail, and add the scalar fields to `shipmentDetailInclude`, or use a separate object that uses `include` with all fields. Simpler: just use `include: shipmentDetailInclude` and rely on Prisma returning all scalar fields by default when `include` is used (no explicit `select`). Map with `mapShipmentDetail`.
- Return `null` if not found.

`create(data: CreateShipmentData): Promise<Shipment>`
- `prisma.shipment.create({ data: { ...data, status: 'Pending' }, select: shipmentListSelect })`.
- Return `mapShipment(row)`.
- FK violations (P2003) should NOT be caught here — they are caught at the service level via explicit lookups.

`updateStatus(id: number, data: UpdateShipmentStatusData): Promise<Shipment>`
- `prisma.shipment.update({ where: { id }, data: { status, ...(shippedAt !== undefined && { shippedAt }), ...(deliveredAt !== undefined && { deliveredAt }) }, select: shipmentListSelect })`.
- Catch `Prisma.PrismaClientKnownRequestError` with `err.code === 'P2025'` → throw `ShipmentNotFoundError()`.

```typescript
// ─────────────────────────────────────────────────────────────────────────────
// Shipment validators
// ─────────────────────────────────────────────────────────────────────────────
const SHIPMENT_STATUSES = [
  'Pending',
  'Shipped',
  'InTransit',
  'Delivered',
  'Failed',
  'Returned',
] as const;

export function validateShipmentCreateData(data: Record<string, unknown>): void {
  const customerOrderId = data['customerOrderId'];
  if (customerOrderId === undefined || customerOrderId === null) {
    throw new ValidationError("Field 'customerOrderId' is required");
  }
  if (!Number.isInteger(customerOrderId) || (customerOrderId as number) < 1) {
    throw new ValidationError("Field 'customerOrderId' must be a positive integer");
  }

  const supplierOrderId = data['supplierOrderId'];
  if (supplierOrderId !== undefined && supplierOrderId !== null) {
    if (!Number.isInteger(supplierOrderId) || (supplierOrderId as number) < 1) {
      throw new ValidationError("Field 'supplierOrderId' must be a positive integer");
    }
  }

  const carrier = data['carrier'];
  if (carrier !== undefined && carrier !== null && carrier !== '') {
    if (typeof carrier === 'string' && carrier.length > 100) {
      throw new ValidationError("Field 'carrier' must not exceed 100 characters");
    }
  }

  const trackingNumber = data['trackingNumber'];
  if (trackingNumber !== undefined && trackingNumber !== null && trackingNumber !== '') {
    if (typeof trackingNumber === 'string' && trackingNumber.length > 100) {
      throw new ValidationError("Field 'trackingNumber' must not exceed 100 characters");
    }
  }

  const trackingUrl = data['trackingUrl'];
  if (trackingUrl !== undefined && trackingUrl !== null && trackingUrl !== '') {
    if (typeof trackingUrl === 'string' && trackingUrl.length > 500) {
      throw new ValidationError("Field 'trackingUrl' must not exceed 500 characters");
    }
  }
}

export function validateShipmentStatusUpdate(data: Record<string, unknown>): void {
  const status = data['status'];
  if (status === undefined || status === null || status === '') {
    throw new ValidationError("Field 'status' is required");
  }
  if (!SHIPMENT_STATUSES.includes(status as (typeof SHIPMENT_STATUSES)[number])) {
    throw new ValidationError(
      `Field 'status' must be one of: ${SHIPMENT_STATUSES.join(', ')}`
    );
  }
}
```

### Key implementation notes

- `validateShipmentStatusUpdate` only validates that `status` is a valid `ShipmentStatus` value. It does NOT enforce transitions — that is the service's job.
- No imports needed — `ValidationError` is already defined in this file.
- Append at end of file to avoid disrupting existing exports.

**Class `ShipmentService`:**

Constructor: `constructor(private readonly shipmentRepository: IShipmentRepository) {}`

**`listShipments(filters: ShipmentFilters): Promise<ShipmentListResult>`**
- Delegates directly to `this.shipmentRepository.findAll(filters)`.

**`getShipmentById(id: number): Promise<Shipment>`**
- `const shipment = await this.shipmentRepository.findById(id);`
- If `null`: throw `new ShipmentNotFoundError()`.
- Return `shipment`.

**`createShipment(input: Record<string, unknown>): Promise<Shipment>`**
- Call `validateShipmentCreateData(input)`.
- Extract: `customerOrderId`, `supplierOrderId`, `carrier`, `trackingNumber`, `trackingUrl`.
- Use `prisma.$transaction` for atomicity:
  - Look up `customerOrder` via `tx.customerOrder.findUnique({ where: { id: customerOrderId }, select: { id: true } })`. If null → throw `new CustomerOrderNotFoundError()`.
  - If `supplierOrderId` provided (not null/undefined):
    - Look up `supplierOrder` via `tx.supplierOrder.findUnique({ where: { id: supplierOrderId }, select: { id: true, trackingNumber: true, trackingUrl: true } })`. If null → throw `new SupplierOrderNotFoundError()`.
    - Pre-fill `trackingNumber` from supplier order if not provided in input (`trackingNumber ?? supplierOrder.trackingNumber ?? null`).
    - Pre-fill `trackingUrl` from supplier order if not provided in input (`trackingUrl ?? supplierOrder.trackingUrl ?? null`).
  - Call `tx.shipment.create({ data: { customerOrderId, supplierOrderId: supplierOrderId ?? null, carrier: carrier ?? null, trackingNumber: resolvedTrackingNumber, trackingUrl: resolvedTrackingUrl, status: 'Pending' } })`.
  - Return `new Shipment({ ...row })`.
- Note: `carrier` is NOT pre-filled from supplier order (supplier order has no `carrier` field in schema).

**`updateShipmentStatus(id: number, input: Record<string, unknown>): Promise<Shipment>`**
- Call `validateShipmentStatusUpdate(input)`.
- Extract `newStatus = input['status'] as ShipmentStatus`.
- Use `prisma.$transaction`:
  - Look up existing via `tx.shipment.findUnique({ where: { id }, select: { id: true, status: true } })`. If null → throw `new ShipmentNotFoundError()`.
  - Call `isValidShipmentTransition(existing.status as ShipmentStatus, newStatus)`. If false → throw `new ShipmentStatusTransitionInvalidError(...)` with message `Cannot transition shipment from ${existing.status} to ${newStatus}`.
  - Compute timestamp side effects:
    - `shippedAt = newStatus === 'Shipped' ? new Date() : undefined`
    - `deliveredAt = newStatus === 'Delivered' ? new Date() : undefined`
  - Call `this.shipmentRepository.updateStatus(id, { status: newStatus, shippedAt, deliveredAt })`.
  - Return updated shipment.
- Note: Use `this.shipmentRepository.updateStatus` for the actual write (not direct prisma) — the repo handles P2025.

### Key implementation notes

- Mixing `prisma` (for FK checks) and `this.shipmentRepository` (for writes) mirrors the `refundService` pattern exactly.
- `supplierOrderId` pre-fill is one-directional: reading from supplier order to pre-fill shipment, never writing back (Design Decision D1).
- `ShipmentStatusTransitionInvalidError` has status 400, not 422 — aligns with spec error code table.
- The service does NOT auto-update `CustomerOrder.fulfillmentStatus` on shipment delivery — this is intentional (Non-Goal in design.md, deferred to future KAN ticket).

```typescript
export interface ShipmentResponseDTO {
  id: number | undefined;

    supplierOrderId: shipment.supplierOrderId,

```

### Key implementation notes

- This serializer is thin — the `Shipment` domain model already carries all data including optional relation summaries.
- For the list endpoint (`GET /`), the controller can return `result` directly (same as refund pattern) without the serializer, since list items don't include relations.
- For `getShipmentById`, the controller calls `serializeShipment(shipment)` to produce a consistent DTO shape.
- If in practice the domain model's JSON serialization is sufficient (no field renaming needed), the serializer can be simplified to a direct pass-through. The value here is explicit DTO typing.

## Critical notes for implementer

1. **`include` vs `select` in Prisma**: `findById` uses `include` (not `select`) for the detail query. When `include` is specified, Prisma returns all scalar fields automatically. Do not combine `include` and `select` on the same query level — it causes a TypeScript error.

2. **`ShipmentStatusTransitionInvalidError` status = 400**, not 422. This differs from refund and supplier-order patterns where transition errors are 409/422. The spec explicitly states 400.

3. **Pre-fill logic**: Only `trackingNumber` and `trackingUrl` are pre-filled from supplier order. `carrier` is NOT (supplier order has no `carrier` field). Pre-fill only applies when the field is not explicitly provided in the request body.

4. **`pageSize` default = 20**, cap = 100. Uses `pageSize` key (not `limit`) in `ShipmentListResult` — aligns with the newer `SupplierOrderListResult` convention, not the older `RefundListResult` convention.

5. **No `fulfillmentStatus` update**: Delivering a shipment does NOT auto-update `CustomerOrder.fulfillmentStatus`. This is by design (MVP manual control). Do NOT add this logic.

6. **Validator test file name**: `validator.shipment.test.ts` (follows the `validator.<domain>.test.ts` convention seen in existing tests).

7. **`errorHandler.ts` must be updated** before running integration tests — otherwise `ShipmentNotFoundError` and `ShipmentStatusTransitionInvalidError` fall through to the generic 500 handler.
