## Context

The storefront (`frontend/`) is a Create React App SPA rendered client-side with `react-router-dom`, split into a `StorefrontLayout` tree (public/customer routes: `/catalog`, `/catalog/:id`, `/pages/:slug`, `/login`, `/cart`, `/checkout`, `/account/*`, etc.) and an admin `Layout` tree (`/products`, `/categories`, `/customers`, `/customer-orders`, `/supplier-orders`, `/shipments`, `/return-requests`, `/refunds`, all behind `RequireAdminAuth`, plus `/admin/login`). Today every route shares one static `<title>`/meta description from `frontend/public/index.html`, and `frontend/public/robots.txt` has no `Disallow` rules and no `Sitemap:` reference. The backend (`backend/src`, Express + Prisma) already exposes public, read-only endpoints under `/api/public/...` (e.g. products, categories) with supplier fields excluded at the Prisma `select` layer — the new sitemap endpoint follows this same pattern.

## Goals / Non-Goals

**Goals:**
- Give every public storefront route (`/catalog`, `/catalog/:id`, `/pages/:slug`) a distinct, accurate `<title>`, meta description, canonical URL, Open Graph/Twitter tags, and (on PDP) `Product` + `BreadcrumbList` JSON-LD.
- Ensure every non-public route (auth, cart, checkout, account, and all admin pages) emits `noindex, nofollow` so it never appears in search results, without changing its functional behavior.
- Make `robots.txt` and a dynamically generated sitemap reflect the real, current catalog and content pages.
- Do this without touching rendering strategy, routing structure, or business logic.

**Non-Goals:**
- SSR/SSG/prerendering migration.
- Locale-segmented URLs / `hreflang`.
- Slug-based PDP routing (`/catalog/:slug`).
- Any change to admin authentication or business workflows beyond adding a `noindex` meta tag.

## Decisions

**1. `react-helmet-async` for per-route `<head>` management.**
Chosen over `react-helmet` (unmaintained, has memory-leak issues with concurrent rendering) and a custom `useEffect`-based `document.title` hack (doesn't support `<meta>`/`<link>`/JSON-LD cleanly, no SSR-readiness if the app ever migrates later). `react-helmet-async` is the de facto standard for CRA-style SPAs, supports a `HelmetProvider` context (needed for correctness under React 18/19 concurrent features), and keeps the door open for SSR without a rewrite.

**2. One shared `Seo` component, not per-page ad-hoc `<Helmet>` blocks.**
A single `frontend/src/components/storefront/Seo.tsx` component takes `title`, `description`, `canonicalPath`, `image`, `noindex`, `jsonLd` props. Alternative considered: let each page compose `<Helmet>` directly. Rejected because the noindex convention must be applied consistently across ~20 routes; a shared component makes "did we forget noindex on an admin page" a single code-review checkpoint instead of 20 independent checks.

**3. Sitemap generated by the backend, not a static file.**
`frontend/public/sitemap.xml` would immediately go stale since products/categories are DB-driven. A new `GET /api/public/sitemap.xml` endpoint queries `Active`-status products, categories, and the fixed list of content-page slugs, and renders XML with `<lastmod>` from `updatedAt`. Alternative considered: build-time static generation (e.g., in `npm run build`) — rejected because catalog data changes independently of deploys, causing stale sitemaps between releases.

**4. `noindex` is a prop on the shared component, not a route-level wrapper.**
Each page passes `noindex` explicitly (default `false`) so it is visible directly in the page component alongside its other SEO props, rather than being inferred from route path — inferring from path is fragile if routes are renamed or added later without updating a separate allowlist.

**5. Admin route noindex list is derived from the actual `App.tsx` routes, not assumed `/admin/*`.**
Verified against `frontend/src/App.tsx`: only `/admin/login` is under `/admin`; the rest of the admin panel (`/products`, `/categories`, `/customers`, `/customer-orders`, `/supplier-orders`, `/shipments`, `/return-requests`, `/refunds`, and their `:id` variants) lives at root-level paths guarded by `RequireAdminAuth`. `robots.txt` and the `noindex` audit must use this real list, not a `/admin/*` prefix assumption.

## Risks / Trade-offs

- **[Risk]** Manually adding `<Seo>` to ~20 pages risks missing one, leaving a private route indexable. → **Mitigation:** add `Seo noindex` at the `Layout`/`StorefrontLayout` wrapper level as a defensive default for admin, and require each new public page to explicitly opt into public metadata; cover with an e2e assertion (`frontend/cypress` or `e2e/`) that checks a representative admin and account page for the `noindex` meta tag.
- **[Risk]** Googlebot renders JS but other crawlers/social-preview bots may not, so Open Graph tags injected client-side may not appear in all link previews. → **Mitigation:** documented as a known limitation in Non-Goals; flagged as a Phase 2 SSR/prerendering follow-up, not silently accepted as "done."
- **[Risk]** Sitemap endpoint could grow slow or return excessive data as the catalog scales. → **Mitigation:** reuse the existing paginated product-query pattern from `/api/public/products`; no sitemap pagination/index needed at current data volume, but note the threshold in the endpoint's code comments for future revisit.
- **[Trade-off]** The static `<title>` in `frontend/public/index.html` remains as a pre-hydration fallback: React 19's native document-metadata hoisting (used by `react-helmet-async`'s React 19 code path) correctly prioritizes the React-rendered `<title>` after mount. However, empirical testing during implementation showed this hoisting does **not** remove pre-existing static `<meta>` tags the way it does for `<title>` — a static `meta[name="description"]` in `index.html` persisted alongside each page's own description after hydration, producing two conflicting description tags in the final DOM (a real SEO regression, since some crawlers/parsers take the first match). **Resolution:** the static `meta[name="description"]` was removed from `index.html`; every storefront route already renders its own complete `Seo` (verified: 100% route coverage), so there is no gap in description coverage after hydration — the only trade-off is a brief no-description window for non-JS-executing clients before React mounts. The defensive fallback `<Seo>` originally planned for `StorefrontLayout` (design decision above, task 2.4) was also removed for the same reason: it duplicated `og:title`/`twitter:title` against each page's own values, since React 19's hoisting does not dedupe `<meta>` tags across independently-mounted `Helmet` instances either. All storefront routes render their own `Seo` directly instead.

- **[Risk]** `frontend/public/robots.txt`'s `Sitemap:` directive hardcodes the current AWS API Gateway invoke URL (`g54xfd8lja...`). CRA only templates `index.html` at build time (via `%PUBLIC_URL%`), not other files under `public/`, so this value is **not** kept in sync with `REACT_APP_API_BASE_URL` automatically. If the API Gateway is ever recreated (not a normal `serverless deploy` update, but a stack teardown/recreate) this URL would go stale silently — a crawler would fail to fetch the sitemap with no build-time or runtime error surfaced. → **Mitigation (adopted):** added a `#` comment in `robots.txt` itself as a manual-update reminder cross-referencing this note. → **Deferred follow-up:** template `robots.txt` at build/deploy time (e.g. a small Node script run before `npm run build`, or a CloudFront/API Gateway rewrite so the sitemap is served same-origin) — track as a separate OpenSpec change if the API Gateway URL changes disruptively often in practice.
- **[Risk]** Product `Product` JSON-LD initially selected price/availability from two different variant sources (`priceVariant`, used for the cart UI, vs. a separate `hasActiveVariant` check), which could show a price from a non-Active variant labeled `InStock`. → **Mitigation (adopted):** structured data now derives `sku`/`offers` from a single Active-variant lookup, omitting `offers` entirely when no Active variant exists, so price and availability never disagree.
- **[Risk]** JSON-LD is serialized with `JSON.stringify` and injected as a `<script>` child; product `name`/`description` containing the literal string `</script>` could break out of the script tag. → **Mitigation (adopted):** `Seo.tsx` replaces every less-than sign in the serialized JSON with its Unicode escape before injecting it, so `</script>` can never appear literally in the rendered markup.

## Migration Plan

1. Add `react-helmet-async` dependency and wrap `App.tsx` with `HelmetProvider` (non-breaking, additive).
2. Introduce `Seo.tsx` and wire it into public pages first (`CatalogPage`, `ProductPage`, `ContentPage`), verifying titles/canonicals render correctly in the browser dev tools before touching private routes.
3. Add `Seo noindex` to all non-public pages (auth, cart, checkout, account, admin).
4. Update `robots.txt` and add the backend sitemap endpoint; verify `GET /api/public/sitemap.xml` returns valid XML via curl.
5. No database migration, no breaking API changes, no rollback risk beyond reverting the commit — every change is additive metadata/markup.
