## Critical Design Decisions (Frontend)

1. **Two-step checkout**: Step 1 = contact info + address + coupon → POST to backend → get `clientSecret`. Step 2 = Stripe Elements payment form. Cart is cleared **after step 1** (order created), not after payment, so a failed/retried payment doesn't re-enter the cart.

2. **`loadStripe` called on CheckoutPage mount** — not lazily at payment step. This gives Stripe.js time to load while the user is filling address fields, eliminating a UX delay.

3. **`redirect: 'if_required'`** on `stripe.confirmPayment` — avoids a full-page redirect for standard card payments; the browser only redirects if the payment method absolutely requires it (e.g., some 3DS flows). Cards and most methods succeed inline.

4. **Polling lives in `OrderConfirmationPage`** — the page polls `GET /api/public/account/orders` (authenticated) every 2 s for up to 30 s. Guest checkouts have no auth token: the polling call will 401 and the interval clears at 30 s → timeout message. This is acceptable for MVP.

5. **3DS redirect detection** — `OrderConfirmationPage` also inspects the `redirect_status` URL query param appended by Stripe after a 3DS redirect. If present, it starts polling even without `location.state.needsPaymentPolling`.

6. **`PaymentForm` is a separate component** — it must be a child of `<Elements>` to use `useStripe()` and `useElements()`. Extracting it keeps `CheckoutPage` from needing those hooks directly.

**Function 1 — `getStripeConfig`:**

```ts
    `${API_BASE}/api/public/payments/config`
  );
```

**Function 2 — `getOrderPaymentStatus`:**
```ts
export async function getOrderPaymentStatus(
  orderNumber: string
): Promise<string | null> {
  const token = getCustomerAccessToken();
  if (!token) return null;
  const res = await axios.get<{
    data: { items: Array<{ orderNumber: string; paymentStatus: string }> };
  }>(`${API_BASE}/api/public/account/orders`, {
    headers: { Authorization: `Bearer ${token}` },
    withCredentials: true,
  });
  const found = res.data.data.items.find((o) => o.orderNumber === orderNumber);
  return found?.paymentStatus ?? null;
}
```
- Returns `null` for unauthenticated users (no token).
- Returns `null` if the order is not found in the list.
- Callers treat `null` as "status unknown, keep polling or time out".

## File 4: `frontend/src/components/storefront/PaymentForm.tsx` (NEW)

### What to create

A child component of `<Elements>` that renders the Stripe `<PaymentElement>` and handles payment submission.

**Reason for separate component:** `useStripe()` and `useElements()` hooks from `@stripe/react-stripe-js` must be called inside a component that is a descendant of `<Elements>`. `CheckoutPage` mounts `<Elements>` conditionally, so `PaymentForm` must be its own component.

### Props interface

```ts

  orderNumber: string;
  onSuccess: () => void;
  onError: (message: string) => void;
}
```

### State

```ts
const [paying, setPaying] = useState(false);
const [paymentError, setPaymentError] = useState('');
```

### Hooks

```ts
const stripe = useStripe();
const elements = useElements();
```

### Handler — `handlePay`

```ts
const handlePay = async (e: React.FormEvent) => {
  e.preventDefault();
  if (!stripe || !elements) return;
  setPaymentError('');
  setPaying(true);
  try {
    const { error, paymentIntent } = await stripe.confirmPayment({
      elements,
      confirmParams: {
        return_url: `${window.location.origin}/order-confirmation/${orderNumber}`,
      },
      redirect: 'if_required',
    });
    if (error) {
      setPaymentError(error.message ?? 'Payment failed. Please try again.');
      onError(error.message ?? 'Payment failed.');
    } else if (paymentIntent?.status === 'succeeded') {
      onSuccess();
    } else {
      // requires_action or redirect scenario — Stripe.js handles the redirect
      // This branch is only reached on an unexpected status; treat as pending
      onSuccess();
    }
  } catch {
    setPaymentError('An unexpected error occurred. Please try again.');
    onError('Unexpected error during payment.');
  } finally {
    setPaying(false);
  }
};
```

### JSX structure

```tsx
return (

    <PaymentElement />
    {paymentError && (
      <Alert variant="danger" className="mt-3" data-testid="payment-error">

    )}
    <Button
      type="submit"
      disabled={!stripe || paying}
      className="w-100 mt-3"
      data-testid="btn-pay"
    >
      {paying ? 'Processing…' : 'Pay now'}
    </Button>
  </Form>
);
```

### New state


### 3DS redirect detection

```ts
const [searchParams] = useSearchParams();
const redirectStatus = searchParams.get('redirect_status');   // 'succeeded' | 'failed' | null
```

### Determine whether to poll

```ts
// Read from location state (normal flow)
const needsPolling = (location.state as { needsPaymentPolling?: boolean } | null)
  ?.needsPaymentPolling === true;

// OR arrived via 3DS redirect
const arrivedVia3ds = redirectStatus !== null;

const shouldPoll = needsPolling || arrivedVia3ds;
```

### Polling `useEffect`

Constants:
```ts
const POLL_INTERVAL_MS = 2000;
const POLL_MAX_MS = 30000;
```

```ts
useEffect(() => {
  if (!shouldPoll || !orderNumber) return;

  setPollingStatus('polling');
  let elapsed = 0;

  const interval = setInterval(async () => {
    elapsed += POLL_INTERVAL_MS;
    try {
      const status = await getOrderPaymentStatus(orderNumber);
      if (status === 'Paid') {
        setPollingStatus('paid');
        clearInterval(interval);
        return;
      }
    } catch {
      // Network error — continue polling
    }
    if (elapsed >= POLL_MAX_MS) {
      setPollingStatus('timeout');
      clearInterval(interval);
    }
  }, POLL_INTERVAL_MS);

  return () => clearInterval(interval);
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [shouldPoll, orderNumber]);
```

**Note:** The dependency array intentionally omits `getOrderPaymentStatus` (stable import reference). Use the eslint-disable comment only on this line.

```ts
    { wrapper: ({ children }) => <LocationStateWrapper state={{ needsPaymentPolling: true }}>{children}</LocationStateWrapper> }
  );

  act(() => { jest.advanceTimersByTime(2000); });

  expect(await screen.findByTestId('payment-confirmed')).toBeInTheDocument();
  jest.useRealTimers();
});
```

**Note:** Use `act()` to wrap `jest.advanceTimersByTime` so React batches state updates from the timer callback. Then use `findByTestId` for the async assertion (not `waitFor + getByTestId`).

## Important Notes for the Implementer

### Cart clearing timing

`clearCart()` is called in `CheckoutPage.handleSubmit` **immediately after step 1 succeeds** (order created), NOT after payment. This is intentional:

- The order exists in the database; the cart items are now committed.
- If payment fails and the user retries, they retry the same PaymentIntent — the cart is not needed.
- If the user navigates away after step 1 but before paying, the order stays in `PendingPayment` on the server. This is acceptable for MVP.

### No `App.tsx` changes needed

The routes `/checkout` and `/order-confirmation/:orderNumber` already exist. The `OrderConfirmationPage` and `CheckoutPage` are already imported. Do not add new routes.

### Guest checkout polling

Guests do not have an auth token. `getOrderPaymentStatus` returns `null` immediately (no API call). After 30 s of polling returning `null`, the timeout message shows. This is acceptable per the spec: "Payment processing — check your email for confirmation."

### `@stripe/react-stripe-js` mock hoisting problem

`jest.mock(...)` factories are hoisted to the top of the test file. Any variable defined in the test file's outer scope (like `const mockStripe = ...`) is **not yet defined** when the factory runs. For this reason:

- Do NOT try to reference outer `jest.fn()` variables inside the `@stripe/react-stripe-js` factory.
- Instead, **mock the `PaymentForm` component itself** in tests for `CheckoutPage`. This avoids the hoisting problem entirely and makes tests cleaner.
- For tests of `PaymentForm` itself, use `jest.mock('@stripe/react-stripe-js')` with `jest.requireMock` to get the auto-mocked module and control `useStripe`/`useElements` return values per test.

### 3DS redirect handling edge case

When Stripe redirects the browser after 3DS, the `OrderConfirmationPage` is hit via a real URL like:
```
/order-confirmation/ORD-000001?payment_intent=pi_xxx&payment_intent_client_secret=xxx&redirect_status=succeeded
```
`location.state` will be `null` (browser navigated via redirect, not React Router `navigate()`). The `redirect_status=succeeded` query param is the signal to start polling. The `useSearchParams()` hook handles this transparently.
